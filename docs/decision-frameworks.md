# 意思決定・アイデア創出フレームワーク集

> LLM向けに最適化された実践的フレームワーク集。具体例とスコアリング基準を含む。

## フレームワーク選択ガイド {#selection-guide}

| 状況 | 推奨フレームワーク | 理由 |
|------|------------------|------|
| タスク数が多い（10+件） | Eisenhower Matrix | 高速な粗選別が可能 |
| 技術的な優先順位付け | ICE Score | 開発工数（Ease）を考慮 |
| プロダクト機能の判断 | RICE Score | ユーザーリーチを重視 |
| 緊急の意思決定 | Eisenhower Matrix | 5分で判断可能 |
| 不確実性が高い | ICE Score (Confidenceを重視) | リスク評価が容易 |
| 問題の本質理解 | First Principles | 常識を疑い根本から考える |

---

## 🎯 意思決定フレームワーク {#decision-frameworks}

### 1. Eisenhower Matrix（アイゼンハワーマトリクス） {#eisenhower}

**緊急度と重要度で4象限に分類**:

```
              緊急           緊急でない
重要      1. 今すぐやる    2. スケジュールする
重要でない  3. 委任する      4. やらない
```

**適用例**:
- タスクA: 象限2（重要だが緊急でない）→ スケジュール実施
- タスクB: 象限4（重要でも緊急でもない）→ 削除

**判断基準**:
- **重要**: ゴール達成に直接影響、技術的負債の解消、セキュリティ対応
- **緊急**: 期限が迫っている、ブロッカーになっている、ユーザー影響大

---

### 2. ICE Score（Impact, Confidence, Ease） {#ice-score}

**各タスクを3軸で評価**:

```
ICE Score = (Impact × Confidence × Ease) / 3
```

**評価軸**:
- **Impact（影響度）**: 1-10点
- **Confidence（確信度）**: 0-100%（0.0-1.0で計算）
- **Ease（容易さ）**: 1-10点（10=最も容易）

**適用例**:

```yaml
高速化機能-A:
  Impact: 7        # DX向上
  Confidence: 90%  # 確実に効果あり
  Ease: 10         # 10分で実装
  ICE Score: (7 × 0.9 × 10) / 3 = 21.0 ⭐⭐⭐

自動化機能-B:
  Impact: 3        # リグレッション防止
  Confidence: 30%  # 事例なし
  Ease: 1          # 8-11時間
  ICE Score: (3 × 0.3 × 1) / 3 = 0.3 ❌
```

**判断基準**:
- **20以上**: 最優先（即実施）
- **10-20**: 高優先（スケジュール実施）
- **5-10**: 中優先（バックログ管理）
- **5未満**: 低優先（削除候補）

**Impact評価基準**:
- **10**: システム全体に影響、ビジネスクリティカル
- **7-9**: 主要機能への影響、DX大幅向上
- **4-6**: 限定的だが明確な改善
- **1-3**: 軽微な改善、限定的な影響

**Ease評価基準**:
- **10**: 10-30分（設定変更、ドキュメント追記）
- **7-9**: 1-3時間（小規模機能追加）
- **4-6**: 1日-1週間（中規模機能実装）
- **1-3**: 1週間以上（大規模リファクタリング）

---

### 3. RICE Score（Reach, Impact, Confidence, Effort） {#rice-score}

**より詳細な優先順位付け**:

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

**評価軸**:
- **Reach（リーチ）**: 影響を受けるユーザー数/利用頻度（期間内）
- **Impact（影響度）**: 0.25/0.5/1/2/3（1人あたりの影響）
- **Confidence（確信度）**: 0-100%
- **Effort（工数）**: 人月（例: 10分 = 0.003人月）

**適用例**:

```yaml
高速化機能-A:
  Reach: 100        # 週100回実行
  Impact: 1         # 中程度の改善
  Confidence: 90%
  Effort: 0.003     # 10分
  RICE: (100 × 1 × 0.9) / 0.003 = 30,000 ⭐⭐⭐

並列化機能-C:
  Reach: 10         # 大規模プロジェクトのみ
  Impact: 2         # 高速化
  Confidence: 80%
  Effort: 0.05      # 2-3時間
  RICE: (10 × 2 × 0.8) / 0.05 = 320 ⚠️
```

**Impact評価基準**:
- **3**: ゲームチェンジャー（劇的な改善）
- **2**: 高い影響（顕著な改善）
- **1**: 中程度の影響（明確な改善）
- **0.5**: 低い影響（小さな改善）
- **0.25**: 最小限の影響（微細な改善）

**判断基準（相対評価）**:
- 上位20%: 最優先
- 上位20-50%: 高優先
- 上位50-80%: 中優先
- 下位20%: 削除候補

---

### 4. Confidence評価の根拠基準 {#confidence-criteria}

**判定基準**:
- **90-100%**: 過去に同様の成功事例あり、または実データあり
- **70-90%**: 類似事例から推測可能
- **50-70%**: 仮説段階だが論理的根拠あり
- **50%未満**: 不確実性が高い（Spikeで検証推奨）

**適用例**:
```yaml
高速化機能-A:
  Confidence: 90%
  根拠: |
    - 関連ドキュメント・実装が既に存在
    - 類似機能の実装で技術的難易度は既知
    - 過去の類似実装で成功実績あり

新技術導入-D:
  Confidence: 40%
  根拠: |
    - 新技術スタックのため事例なし
    - 技術調査（Spike）が必要
    → Spike（1-2日）で70%以上に引き上げてから判断
```

## 🧠 思考プロセスフレームワーク {#thinking-frameworks}

### 5. First Principles Thinking（第一原理思考） {#first-principles}

**常識を疑い、根本原理から考え直す**:

**ステップ**:
1. 前提を明確化
2. 前提を分解
3. 根本原理から再構築

**適用例**:

```yaml
前提: "テストがあれば品質が上がる"

分解:
  - テストが品質向上に寄与するのはいつ？
    → リグレッションが頻繁に起こる場合
  - 現状はリグレッションが起きているか？
    → 過去1年間0件

根本原理から再構築:
  → リグレッションが起きてから書けば良い（YAGNI）
  → 今は実装しない
```

**適用場面**:
- 「常識」「ベストプラクティス」を疑うとき
- 既存の解決策が複雑すぎるとき
- イノベーションが必要なとき

---

### 6. SCAMPER {#scamper}

**既存のものを組み替えてアイデア創出**:

- **S**ubstitute（代替）: 別の技術・方法に置き換えられないか？
- **C**ombine（結合）: 複数の機能を統合できないか？
- **A**dapt（適応）: 他の分野の手法を応用できないか？
- **M**odify（修正）: サイズ・形・属性を変更できないか？
- **P**ut to another use（転用）: 別の用途に使えないか？
- **E**liminate（削除）: 不要な要素を削除できないか？
- **R**everse（逆転）: 逆の発想はできないか？

**適用例**:
```
課題: 自動化パイプラインが遅い

S: プラットフォームAからプラットフォームBに代替？
C: 処理1 + 処理2 を1ステップに統合？
A: 他領域のインクリメンタル手法を応用？
M: 並列度を2 → 4に変更？
P: キャッシュ機構をローカル環境にも転用？
E: 不要な検証ステップを削除？
R: 軽い処理を先に実行して重い処理を後にする？
```

---

### 7. Jobs-to-be-Done（JTBD） {#jtbd}

**ユーザーが「雇用する仕事」の観点で問題を捉える**:

**原則**: 「ドリルではなく穴が欲しい」

**質問テンプレート**:
- ユーザーは何を達成しようとしているのか？
- なぜそれを達成したいのか？
- 現在どのように達成しているのか？
- 何が障害になっているのか？

**適用例**:
```
ユーザー要望: "高速な検証ツールが欲しい"

JTBD分析:
  達成したいこと: 開発フローを中断せずに作業を続ける
  理由: 検証待ち時間で集中力が途切れる
  現状: 検証コマンドで30秒待つ
  障害: 全ファイルが毎回検証される

解決策（ドリルではなく穴）:
  → Watch mode + 変更ファイルのみ検証実行
  → 高速なツール導入は手段の1つに過ぎない
```

## 🚀 実践的ワークフロー {#practical-workflows}

### パターンA: 大量タスクの優先順位付け {#workflow-a}

```
1. Eisenhower Matrix で粗選別（象限1-2に絞る）
2. ICE Score で詳細評価
3. First Principles で上位3件を検証（本当に必要か？）
4. 実施
```

**適用例**:
```
15件のタスク → Eisenhower → 6件に絞る
→ ICE Score → 上位3件（21.0, 18.5, 15.2）
→ First Principles → 21.0, 18.5 のみ実施（15.2は不要と判明）
```

---

### パターンB: 新機能の意思決定 {#workflow-b}

```
1. First Principles で問題の本質を理解
2. RICE Score で複数案を比較
3. Pre-mortem でリスク洗い出し
4. Build-Measure-Learn で小さく試す
```

**適用例**:
```
要求: "認証機能を強化したい"

First Principles:
  → 本質: 不正アクセス防止、ユーザー体験向上

RICE Score:
  案A（2要素認証）: RICE 450
  案B（パスワードレス）: RICE 320
  案C（OAuth統合）: RICE 280

Pre-mortem:
  案A: 既存ユーザーの反発リスク
  → 段階的展開で軽減

Build-Measure-Learn:
  MVP: メールでのワンタイムコード（1週間で実装）
  Measure: 利用率、セキュリティインシデント
  Learn: 効果確認後に本格実装
```

---

### パターンC: 緊急対応 {#workflow-c}

```
1. Eisenhower Matrix で象限1（緊急かつ重要）を特定
2. ICE Score で Confidence と Ease を重視（速く確実に）
3. 実施
```

**適用例**:
```
本番障害発生:

Eisenhower:
  → 象限1: データベース接続エラー修正

ICE Score（緊急時は Ease と Confidence を重視）:
  案A: 接続プール増やす（Confidence 90%, Ease 10）
  案B: クエリ最適化（Confidence 60%, Ease 3）
  → 案A を即実施
```

## ❌ アンチパターン {#anti-patterns}

### 失敗1: スコアリングの機械的適用 {#anti-pattern-1}

**問題**: ICE Scoreが高いからといって盲目的に実施

**対策**:
- 上位3件は必ずFirst Principlesで再検証
- 「本当にこれが必要か？」を問い直す

**失敗例**:
```
ICE Score 22.5 の「管理画面のダークモード対応」
→ First Principles: 管理画面は内部ツール、利用時間は日中のみ
→ 実は不要（ユーザー要望の誤解）
```

---

### 失敗2: Confidence過大評価 {#anti-pattern-2}

**問題**: 「たぶんできる」を80%で評価

**対策**:
- 過去の類似事例がなければ50%未満
- 不確実性が高い場合はSpike（調査）を先に実施

**失敗例**:
```
新しいフレームワーク導入:
  誤: Confidence 80%（「たぶんいける」）
  正: Confidence 40%（事例なし）
  → Spike 2日で検証 → 70%に引き上げてから判断
```

---

### 失敗3: Effortの楽観視 {#anti-pattern-3}

**問題**: 「30分」が実際は3時間に

**対策**:
- 見積もりに1.5-2倍のバッファを設定
- 過去の実績データを参照

**失敗例**:
```
"設定ファイル1つ変更するだけ"
  見積: 30分（Ease 10）
  実績: 3時間（依存関係の調査、テスト、ドキュメント更新）
  → 次回は Ease 7 で評価（1-3時間）
```

---

### 失敗4: Reachの過大評価 {#anti-pattern-4}

**問題**: 「全ユーザーが使う」と想定したが実際は一部のみ

**対策**:
- 実データ、アナリティクスで検証
- 保守的に見積もる

**失敗例**:
```
新機能「エクスポート機能」:
  見積: Reach 1000（全ユーザー）
  実績: Reach 50（5%のパワーユーザーのみ）
  → RICE Score が20倍過大評価
```

## 📊 スコアリングテンプレート {#templates}

### ICE Score計算テンプレート

```yaml
task_name: "タスク名"
ice_score:
  impact: 7          # 1-10
  confidence: 0.9    # 0.0-1.0
  ease: 10           # 1-10
  score: 21.0        # (7 × 0.9 × 10) / 3
  priority: "最優先"  # 20以上=最優先

根拠:
  impact: "DX向上、週100回使用される"
  confidence: "類似実装で成功実績あり"
  ease: "10分で実装可能（設定変更のみ）"
```

---

### RICE Score計算テンプレート

```yaml
task_name: "タスク名"
rice_score:
  reach: 100         # ユーザー数/利用頻度
  impact: 1          # 0.25/0.5/1/2/3
  confidence: 0.9    # 0.0-1.0
  effort: 0.003      # 人月
  score: 30000       # (100 × 1 × 0.9) / 0.003
  priority: "最優先"  # 上位20%

根拠:
  reach: "週100回実行される"
  impact: "1人あたり中程度の改善"
  confidence: "既存実装の拡張のため確実"
  effort: "10分（0.003人月）"
```

---

### First Principles分析テンプレート

```yaml
task_name: "タスク名"
first_principles:
  前提: "〇〇があれば△△が向上する"

  分解:
    - question: "△△が向上するのはいつ？"
      answer: "□□の場合"
    - question: "現状は□□が起きているか？"
      answer: "過去1年間0件"

  根本原理から再構築:
    結論: "□□が起きてから実装すれば良い（YAGNI）"
    判断: "今は実装しない"
```
